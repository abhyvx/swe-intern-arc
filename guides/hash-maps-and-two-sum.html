Hash maps and Two Sum

Goal: you can invent the O(n) map pattern yourself, not memorize a solution video.

The slow way (and why interviews reject it)

Two Sum: given numbers and a target, return two indices whose values add to target.

Brute force: for each i, scan all j > i. That is about n² comparisons. Fine for n=10. Painful for n=10,000. Interviews want you to notice that.

The idea: remember what you already saw

While scanning left to right, for value x you need target - x to already have appeared. A hash map (Python dict) stores value → index of numbers you have seen.

Average lookup in a dict is O(1). So one pass becomes O(n).

Walkthrough

nums = [2, 7, 11, 15], target = 9

  1. i=0, x=2, need=7. Map empty. Store {2: 0}
  2. i=1, x=7, need=2. Map has 2 at index 0. Return [0, 1]
def two_sum(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        need = target - x
        if need in seen:
            return [seen[need], i]
        seen[x] = i

When you use this

  • Interview Easy/Medium “find pair / complement” problems
  • Counting frequencies (anagrams, duplicates)
  • Real products: caching an id → object lookup instead of scanning a list

How to write it yourself

  1. Say the brute force out loud
  2. Ask: “What would make the inner scan O(1)?”
  3. Name the key you must remember (usually the value, sometimes a sorted tuple)
  4. Code one pass; test a case with duplicate values

How to fix common bugs

  • Using the same element twice → store index after the check, or ensure need’s index ≠ i
  • Returning values instead of indices → read the problem again
  • dict key is list → lists are unhashable; use tuple or string key

Check yourself

Without looking: write Two Sum on paper. Then explain why the map is better than nested loops to a friend (or voice note).
