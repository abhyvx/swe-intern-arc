How C programs run

Read this before you write hello.c. Goal: you can explain compile vs run without hand-waving.

The problem

Your Mac does not understand a .c file as instructions. A .c file is text for humans. The CPU runs machine code. Something has to translate.

The pipeline

  1. Write source: hello.c
  2. Compile with gcc: turns text into an executable binary (often named hello with no extension)
  3. Run the binary: ./hello
gcc -Wall -Wextra -o hello hello.c
./hello

-o hello means “name the output binary hello”. If you skip -o, gcc often writes a.out.

What -Wall -Wextra are for

C will let you write dangerous mistakes that still “compile”. Warnings are free teaching. -Wall -Wextra turns many silent problems into messages. In CMSC 216, clean compiles matter.

main is the entry

#include <stdio.h>

int main(void) {
  printf("hi\n");
  return 0;
}

#include <stdio.h> pulls in the declaration of printf. return 0 means success to the shell.

sizeof

sizeof(int) is how many bytes an int uses on this machine. On modern Macs, int is often 4, pointers often 8. You measure; you do not guess from vibes.

When you use this

  • Every CMSC 216 lab
  • Any systems project that touches C
  • Debugging “I ran the .c file” mistakes (you run the binary)

How to fix common failures

  • command not found: gcc → install Command Line Tools: xcode-select --install (small tools package, not full Xcode app)
  • error: expected ‘;’ → missing semicolon; read the line above the error too
  • permission denied: ./hello → rare; check you are running the binary you just built
  • unused variable warning → delete it or use it; do not ignore warnings in 216
