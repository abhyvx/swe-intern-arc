How to read a codebase you own

Used for Gambit, TerpButler, internships, hackathons. Goal: find a path and defend it.

Do not read every file

Tourism feels productive and teaches little. Pick one user-visible behavior and chase it.

Method

  1. Boot — if it does not run, fix that first or write exact errors
  2. Find UI entry — where the click/page starts
  3. Find API entry — route/handler for that action
  4. Find data/model — where numbers/strings are computed or loaded
  5. Write hops — file::function → next until you have 5+
  6. Prove one value — print/assert an intermediate

What “owning” means in interviews

  • You can whiteboard the hop list
  • You can name one failure mode
  • You can say what you would build next
  • You can distinguish what you wrote vs what you still do not understand (honesty)

When you use this

  • Internship onboarding on a foreign repo
  • Hackathon base selection
  • Resume project defense

How to fix “I am lost”

  • Search for route strings, app.get, @app., fetch(, axios
  • Use GitHub search in the repo
  • Draw User → UI → API → Data on paper before opening more files
