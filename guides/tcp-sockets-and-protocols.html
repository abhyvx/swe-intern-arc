TCP sockets and line protocols

Foundation for NetKV. Goal: you understand accept loops and why we parse text commands.

What TCP gives you

A TCP connection is a reliable bidirectional byte stream between two programs. It does not know what a “message” is. You invent framing (for us: lines ending in newline).

Server loop (mental model)

  1. socket — create endpoint
  2. bind — attach to host/port
  3. listen — mark as accepting
  4. accept — block until a client connects; returns a new socket for that client
  5. recv / send — read/write bytes

Why echo first

Echo proves your accept/recv/send loop before SET/GET logic. If echo fails, do not debug the protocol yet.

Why parse in pure functions

parse("SET a b") returning a structure lets you unit test without opening ports. Real services separate protocol parsing from networking.

When you use this

  • Any networked service
  • Understanding Redis/Postgres at a high level
  • Systems interviews (“how would you design a KV store?”)
  • Debugging internship voice/realtime stacks: same “bytes on a connection” idea

How to fix common bugs

  • Address already in use → old process still bound; kill it or change port
  • Client hangs → server did not send a response newline; or blocked forever on recv
  • Partial reads → TCP can split data; buffering matters as you grow past toy labs
